# 对话式生图网站设计

## 目标

用最低开发成本做一个仅供少量固定用户使用的中文生图网站，核心体验是“像聊天一样发起创作”，支持上传参考图并调用 Right Codes 生图接口。

## 第一版范围

包含：

- 对话式输入区
- 多张参考图上传与预览
- 尺寸选择
- 调用 Right Codes 生图
- 结果图片展示
- 复制提示词
- 下载图片
- 浏览器本地历史记录

不包含：

- 登录
- 数据库
- 云端同步
- 管理后台
- 多模型切换
- 真正的聊天补全能力

## 方案选择

采用 `Next.js + TypeScript + App Router`。

原因：

- 前后端一体，目录最少
- 服务端接口可安全保存 API Key
- 部署简单，后续要上 Vercel 也顺手
- 对空仓库启动成本最低

样式不引入额外 UI 框架，直接使用全局 CSS，降低依赖与理解成本。

## 交互设计

整体采用“编辑刊物风”：

- 左侧：创作档案
- 中间：主创作流
- 底部：固定输入区

主创作流中的一条记录包含：

1. 用户的创作提示词
2. 上传的参考图缩略图
3. 生成中的状态
4. 生成结果大图
5. 操作区：复制提示词、下载、再次生成

## 数据流

前端：

- 维护当前输入内容、尺寸、参考图
- 维护本地会话历史
- 上传文件后转为 `data:image/...` 字符串再提交给服务端

服务端：

- 接收前端提交的 `prompt`、`size`、`images`
- 转发到 Right Codes `POST /v1/images/generations`
- 返回标准化后的结果给前端

## 存储策略

历史记录只存在浏览器 `localStorage`，结构按接口契约建模，不做额外兼容层。

记录字段：

- `id`
- `prompt`
- `size`
- `createdAt`
- `referenceImages`
- `resultImages`
- `status`
- `errorMessage`

## 风险与处理

- Right Codes 生图响应较慢：前端明确展示生成中状态
- 远程图片下载跨域不稳定：增加服务端下载代理路由
- 第一版不接聊天接口：通过对话式 UI 满足体验目标，避免范围膨胀

## 实现顺序

1. 初始化 Next.js 最小骨架
2. 实现页面布局与对话式生图流
3. 接入 Right Codes 生图 API
4. 接入本地历史与下载能力
5. 补充使用说明
